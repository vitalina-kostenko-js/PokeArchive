# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this repo is

A **default scaffold** for Cloudflare Workers + Hono services. The current `src/` is illustration of the pattern, not committed product surface.

## Architecture

The full architectural specification (Layer/Slice/Segment layout, naming conventions, persistence rules, Cloudflare-specific contract, Mode A bootstrap, Mode B extension) lives in the project-local skill at [.claude/skills/worker-structure/](.claude/skills/worker-structure/). Invoke it with `/worker-structure` when the task is "spin up a new worker", "add a module / widget / feature / entity / shared segment", or "audit this project against the pattern".

## Commands

- `yarn dev` — local dev on port 5000.
- `yarn format` — type-check → lint → prettier. Prefer this before declaring work done.
- `yarn cf-typegen` — regenerate `types-<entry>.d.ts` after binding/DO/migration changes.
- `yarn deploy` — default env. For others: `wrangler deploy --env develop|staging|production`.

## Hard rules

- Never hand-edit any `types-<entry>.d.ts` — they are generated by `yarn cf-typegen` (or per-entry script).
- Never put secrets in any `wrangler.<entry>.jsonc` — use `wrangler secret put --env <name> -c wrangler.<entry>.jsonc`.
- Read process env only through `envConfig` from [src/config/](src/config/), never `process.env` directly.
- When adding a binding / Durable Object / migration, mirror it across **all four** env blocks of **every** `wrangler.<entry>.jsonc` that needs it (`local`, `develop`, `staging`, `production`) — there is no inheritance, neither between env blocks nor between entry configs.
