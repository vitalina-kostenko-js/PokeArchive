# Pitfalls & verification

## Common mistakes

**Layer dependency direction.** A feature importing a widget, an entity importing a feature, or a module importing another module's `*.module.ts` are all upward imports. Lift the shared symbol down: shared types into `shared/interfaces/`, shared logic into a feature/widget, shared persistence into `entities/`.

**Single-block DO definition.** Adding a Durable Object binding or migration to one env block only. Top-level and `env.*` do not inherit. The deploy succeeds for the configured env and fails (or silently mis-binds) for the others.

**Missing DO re-export.** Defining a DO class but forgetting `export { <DOClass> } from './app/entities/models'` in `src/app/<entry>.ts`. Wrangler cannot resolve the class and the DO becomes unreachable.

**Direct `@hono/zod-validator` usage.** Validating a route with `@hono/zod-validator`'s `zValidator` instead of the project's `pkg/validator/` wrapper. The 400 response shape diverges from `{ error: 'Bad Request', message }` and consumer error parsing breaks.

**Business logic in `*.module.ts`.** Module files validate input and dispatch — nothing else. Move all logic to `*.service.ts`.

**Missing cache invalidation.** A `POST` / `PATCH` / `DELETE` route that does not invalidate every affected cache key inside `ctx.executionCtx.waitUntil(...)`. Reads serve stale data for the full `cacheControl: max-age` window. The bug is silent and time-delayed.

**Hand-edited `types-<entry>.d.ts`.** The file is generated by `yarn cf-typegen`. Edits are overwritten on the next regeneration.

**Direct `process.env` access.** Read env variables only through `envConfig`. Adding a var means updating the Zod schema in `env.config.ts` *and* `.env.example`.

**Secret in `wrangler.<entry>.jsonc`.** Use `wrangler secret put --env <name>`. Anything in `wrangler.<entry>.jsonc` is checked in.

**Mixed concerns in `shared/`.** A util that calls a service is no longer a util — move it to `shared/services/`. A constant file that imports runtime code is not a constant — split it.

**Skipped barrels.** Every folder ships an `index.ts` and consumers import from the folder. A direct file import in one place becomes a refactor risk later.

## Verification checklist

Before declaring work done, confirm:

**Build & lint**
- [ ] `yarn type-check` passes.
- [ ] `yarn lint` passes (or `yarn format` end-to-end).
- [ ] `yarn dev` boots; `GET /` returns the health text.

**For a new module**
- [ ] Module file is thin (validation + delegation only).
- [ ] Service returns `ctx.json(...)` for every path, including the standard 500 on `try/catch` failure.
- [ ] Each CRUD verb returns the schema-validated shape.
- [ ] List endpoint reflects mutations after invalidation.
- [ ] Unauthenticated requests to `/v1/*` return 401.

**For a new Durable Object**
- [ ] Binding + migration are present in **all four** env blocks of `wrangler.<entry>.jsonc`.
- [ ] Class re-exported from `src/app/<entry>.ts`.
- [ ] `yarn cf-typegen` regenerated `types-<entry>.d.ts`; `ctx.env.<BINDING>` is typed.
- [ ] Accessor helper is exported alongside the class so services do not call `idFromName` directly.

**Layout discipline**
- [ ] Every new folder has an `index.ts` barrel.
- [ ] Imports flow only downward (`grep -R "from '\.\./\.\./modules'" src/app/modules/` returns nothing — modules do not import other modules).
- [ ] Filenames use the correct suffix for the layer (`*.module.ts`, `*.service.ts`, `*.dto.ts`, `*.object.ts` / `*.table.ts` / `*.collection.ts`, `*.constant.ts`, `*.interface.ts`, `*.middleware.ts`, `*.util.ts`).

**Configuration**
- [ ] Every new env var is in the Zod schema in `src/config/env.config.ts` and listed in `.env.example`.
- [ ] No `process.env.<X>` reads outside `env.config.ts`.
- [ ] No secret values in `wrangler.<entry>.jsonc`.
